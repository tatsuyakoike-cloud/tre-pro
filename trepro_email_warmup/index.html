ウォームアップ配信
要件定義・リスト定義

作成日: 2026-06-21 対象: 過去掘り起こしリスト × 初回一斉メール配信 配信期間: 2日間 / 5段階

この資料について（第三者向けサマリー）

本資料は、過去リードを対象とした初回メールナーチャリング配信の設計書です。 別ファイルでリストを管理するのではなく、元マスタースプレッドシートに補助列を追加し、 保存済みフィルター3条件だけで配信対象を抽出できる状態を目指します。

初回大量配信によるレピュテーション低下を避けるため、IPウォームアップの観点から 2日間で5回に分け、配信量を段階的に増やします（約140件 → 約600件）。

自社メール配信ソフト使用 メール重複除外 バウンス・配信停止を恒久除外 配信ステータスで再配信防止
1

要件定義

ビジネス目的・制約・成功条件を整理したもの。開発・配信・営業の各担当が同じ前提で動くための定義です。

1-1. 目的

  • ▶
    ナーチャリング施策の開始 過去の掘り起こしリストを活用し、メール配信でリードとの接点を再開する。
  • ▶
    安全な初回一斉配信 初めての一斉配信のため、IPウォームアップを意識し、少量から段階的に配信量を増やす。
  • ▶
    再現性のあるリスト抽出 配信担当者が毎回同じ条件でリストを取り出せるよう、元マスター上で設計する。

1-2. 前提条件

項目内容
データソース過去の掘り起こしリスト（元マスター）
配信ツール自社開発メール配信ソフト
配信スケジュール2日間 / 5段階（1日目AM・PM、2日目AM・PM前半・PM後半）
リスト管理別ファイルではなく元マスターに補助列を追加
目安総配信数約1,710件（有効リスト件数に応じて比率で調整）

1-3. 考慮すべきリスク

  • !
    レピュテーション低下 一度に大量配信すると、送信ドメイン/IPの評価が下がる可能性がある。
  • !
    バウンス率上昇 古いリストほど無効アドレスが混在しやすく、到達率悪化の原因になる。
  • !
    重複・誤配信 同一メールへの二重配信、配信停止・バウンス済みへの再送を防ぐ必要がある。
  • !
    属人化 担当者ごとに抽出条件がブレると、品質と再現性が失われる。

1-4. 配信優先順位（並び替えルール）

有効リストを以下の順で並べ、上位から配信1〜5に割り当てる。

順位判定軸意図
1確度成約見込みが高いリードを最優先
2電話番号の取得反応後の架電が可能なリードを優先
3決裁者性役職 / BANTコメント / 決裁者レベルで判断
4直近接触の新しさアプローチ履歴・初回商談日・問い合わせ日から算出
5メール有効性形式チェック済みのアドレスのみ
6重複なし同一メールアドレスは1件のみ

1-5. 配信対象の包含・除外条件

すべて満たす場合に配信対象

  • ✓
    メールアドレスが空欄でない
  • ✓
    メール形式が有効
  • ✓
    メール重複判定 = 初回
  • ✓
    配信停止リストに含まれない
  • ✓
    ハードバウンス済みでない
  • ✓
    配信ステータス = 未配信

除外対象

  • ×
    メール空欄・形式不正
  • ×
    重複行（2件目以降）
  • ×
    配信停止依頼あり
  • ×
    過去ハードバウンス
  • ×
    配信済み
  • ×
    明確な配信対象外
2

リスト定義

元マスターに追加する補助列の定義。スプレッドシート実装者・配信担当者が参照する仕様です。

2-1. 参照する既存カラム

カラム用途
確度優先度の最重要指標
担当者電話番号架電可否の判定
役職 / BANTコメント / 決裁者レベル決裁者性の判定
アプローチ履歴 / 初回商談日 / 問い合わせ日直近接触日の算出
メールアドレス配信先・重複除外
配信ステータス未配信 / 配信済み / バウンス等の管理

2-2. 追加する補助列

列名出力値判定ロジック（概要）
メール正規化 文字列 メールアドレスを小文字化・前後空白除去。表記ゆれを統一する。
メール重複判定 初回 / 重複 正規化メールが複数行ある場合、優先順位が最も高い1行のみ「初回」。それ以外は「重複」。
配信可否 配信対象 / 除外 メール有効・重複なし・配信停止/バウンスでない場合に「配信対象」。
確度スコア 数値 確度を数値化（例: S=100, A=80, B=60, C=40, D=20）。並び替え用。
電話OK OK / NG 担当者電話番号が取得済みかつ有効形式。
決裁者OK OK / NG 役職・BANT・決裁者レベルから決裁者または準決裁者と判定。
直近接触日 日付 アプローチ履歴・初回商談日・問い合わせ日の最新日付。
接触経過区分 3ヶ月以内 / 6ヶ月以内 / 12ヶ月以内 / 12ヶ月超 直近接触日からの経過で分類。
優先順位 整数（1〜N） 確度スコア降順 → 電話OK → 決裁者OK → 直近接触日降順で一意に付番。
配信回 配信1〜配信5 / 空欄 配信可否=配信対象の行を優先順位順に並べ、累積比率で5分割。
配信ステータス 下表参照 初期値「未配信」。配信後に手動またはCSV取込で更新。

2-3. 配信回の割り当てロジック

配信可否 = 配信対象 の行のみを対象に、優先順位の昇順（1が最上位）で並べたうえで、累積件数比率で分割する。

配信回累積比率（上限）該当イメージ
配信1〜 8%最優先・最安全層
配信2〜 20%（8%+12%）高優先度層
配信3〜 38%（+18%）中位有望層
配信4〜 65%（+27%）掘り起こし拡大層
配信5〜 100%（+35%）残り全件
実際の件数は有効リスト総数 × 比率で算出する。総数が変動した場合も比率ベースで再計算し、配信回列を更新する。

2-4. 配信ステータス定義

未配信 配信済み バウンス 反応あり 配信停止 除外
ステータス定義次回配信
未配信まだ配信していない対象（フィルター条件を満たす場合）
配信済み配信完了除外
バウンスハードバウンス等で不達恒久除外
反応あり返信・クリック・問い合わせあり営業優先対応。一斉配信からは除外
配信停止配信停止依頼恒久除外
除外形式不正・重複等対象外

2-5. CSV出力項目（配信ツール取込用）

フィルター適用後にエクスポートする主要列。

会社名 / 担当者名 / メールアドレス / 電話番号 / 役職 / 確度 / 直近接触日 / 配信回
3

配信ボリューム設計

2日間・5段階で配信量を増やす。1回目はテスト兼ウォームアップとして最小構成とする。

配信回 タイミング 比率 目安件数 目的
配信11日目 AM約8%約140件初回テスト・バウンス率確認
配信21日目 PM約12%約200件初日追加・問題なければ拡大
配信32日目 AM約18%約310件配信量拡大
配信42日目 PM前半約27%約460件本格配信
配信52日目 PM後半約35%約600件残リスト一括
配信2以降に進む前に、前回配信のバウンス率・エラー・迷惑メール報告・配信停止依頼を確認する。異常があれば次回配信を停止し、リスト・文面・システム設定を見直す。
4

保存フィルター定義

配信担当者は、タイミングに応じて保存済みフィルターを選ぶだけでリストを抽出する。

4-1. 配信用フィルター（5種）

いずれも以下3条件の組み合わせ。配信回の値だけが異なる。

配信可否 = 配信対象 配信回 = 配信N（1〜5） 配信ステータス = 未配信
保存名タイミング配信回想定リストの特徴
配信1_1日目AM1日目 AM配信1最高確度・電話/決裁者あり・接触新しい
配信2_1日目PM1日目 PM配信2配信1に次ぐ高優先度
配信3_2日目AM2日目 AM配信3中位有望・ナーチャリング向け
配信4_2日目PM前半2日目 PM前半配信4確度やや低・接触古めも含む
配信5_2日目PM後半2日目 PM後半配信5配信可能な残り全件

4-2. 管理用フィルター

保存名条件用途
除外_要確認配信可否 = 除外形式不正・重複等の確認
バウンス確認配信ステータス = バウンス不達アドレスの一覧
反応あり確認配信ステータス = 反応あり架電・個別対応優先リスト
配信停止確認配信ステータス = 配信停止恒久除外リストの確認
5

配信担当者の運用フロー

元マスターを開く 補助列・配信回・配信ステータスが最新か確認する。
タイミングに合わせてフィルターを選択 例: 1日目AM →「配信1_1日目AM」
未配信のみであることを確認 配信済み行が混ざっていないか目視チェック。
CSV出力 → 配信ソフトへ取込 会社名・担当者名・メール・電話・役職・確度・直近接触日・配信回を出力。
配信後、結果を元マスターへ反映 配信済み / バウンス / 反応あり / 配信停止 / 要架電 / 要個別対応 を更新。バウンス・配信停止は別シートにも記録。

配信後チェック（次回配信前）

確認項目判断基準
バウンス率想定以上に高くないか
配信エラー自社ソフト側の異常がないか
迷惑メール報告クレーム・報告がないか
配信停止依頼即時ステータス更新したか
返信内容ネガティブ返信・営業機会の有無
到達状況配信ログ上の異常がないか
6

重複防止・恒久除外ルール

  • 1
    メールアドレス単位で重複除外 同一正規化メールは1行のみ配信対象。メール重複判定 = 初回 の行だけを含める。
  • 2
    1リード = 1配信回 配信1〜5のいずれか1つのみ付与。複数回への重複割当は設計しない。
  • 3
    配信後は必ずステータス更新 次回フィルターは必ず 配信ステータス = 未配信 を含める。
  • 4
    配信停止・バウンスは恒久除外 別シート（配信停止リスト / バウンスリスト）にも記録し、配信可否判定に反映する。

決定事項（合意内容）

  • ✓
    別ファイル管理ではなく元マスター上で完結
  • ✓
    保存フィルター3条件で抽出（配信可否 / 配信回 / 配信ステータス）
  • ✓
    5段階ウォームアップ配信（2日間）
  • ✓
    メール単位の重複防止と配信後ステータス更新を必須化
